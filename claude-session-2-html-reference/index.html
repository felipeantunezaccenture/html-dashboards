Understanding Claude's Scope Levels

Claude Code organises all configuration across four distinct levels, from machine-wide admin policies down to project-level team settings. Each level narrows the scope of who and what it affects, and they stack on top of each other so more specific settings take precedence.

Full
scope
Most
specific
System-wide
C:\Program Files\ClaudeCode\
Admin-deployed by IT or DevOps. Applies to every user on this machine. As Accenture employees, we cannot edit this — it is defined and controlled at the Accenture organisation level.
Locked
User Level (Global)
C:\Users\{username}\.claude\
Your personal settings, skills, memory, and more. Applies to all your projects on this machine. Never shared with teammates.
Project Level
Overrides your global defaults for this project.
Session Level
Instructions and context given directly in the current conversation. Only active for this session — not persisted anywhere.

The golden rules of scope

  • If you want to reuse a skill across projects or have Claude remember something about you or how you like to work, store it at the global level. More specific rules and ways of working that differ from project to project should be saved at the project level.
  • Deny always wins — if a tool is denied at any level, it is blocked everywhere. No other level can override a deny

What is a Skill?

Skills are essentially very specific instructions, written in a Markdown format. Claude reads them every time you call the Skill. Write it once, call it forever with /skill-name.

💡

The rule of thumb

If you've typed the same instructions more than twice → it's a Skill.

Markdown: the language between you and Claude

Claude reads Markdown in your prompts and Skill files, and writes it in every response. The better your Markdown, the more precise and consistent Claude's output will be.

Claude reads it

In prompts & Skill files

Any Markdown you write in a prompt or SKILL.md is interpreted by Claude as structure instructions.

Claude writes it

In every response

That's the ## and **bold** you see. You can specify exactly what format you want and Claude will follow it.

Best practices
Keep it under 200 lines

Claude reads the full file on every call. Long = slower + less precise. Under 200 lines keeps it sharp.

Use modular format: one ## section per concern

Separate "when to use", "output format", "rules", and "example". Claude navigates sections, not walls of text.

Be specific in "Output format"

The most common mistake. Vague instructions → vague output. Name every header, bullet style, and field.

Include an example output

Paste a real example of exactly what you want. One good example outperforms three paragraphs of instructions.

Name it for how you'll call it

/meeting-notes is better than /my-note-formatter-v2. Short, obvious, memorable.

Skills vs Agents

Both live in Claude Code, but they are fundamentally different. Skills shape how Claude responds. Agents decide what Claude does next.

"A Skill is a style guide you hand to a writer. An Agent is a contractor: you describe the outcome, they figure out the steps."

Skill ⚡Agent
What it isSaved instructions: shapes how Claude respondsAutonomous Claude: decides what to do next
Who drivesYou: you call it, you stay in controlClaude: it plans and acts independently
File accessNo, reads only its instruction fileYes, reads, writes, creates files
Best forFormatting, tone, templates, output rulesMulti-step tasks: research → draft → format → save
RiskLow, shapes conversation onlyHigher, acts on your files; always review
Example/meeting-notes → applies your format rules"Build me a competitive analysis" → Claude does it all

Calling a Skill & the team library

Two ways to invoke a Skill: explicit is predictable, automatic is convenient. The team library gives everyone the same starting point.

Explicit call

/skill-name [your request]

Type the Skill name at the start of your message. Claude applies it immediately and exactly. Use this when the format really matters.

/meeting-notes  Write up today's standup
/slide-deck  Q3 findings for client review
/branding  Draft the project one-pager
Automatic

Claude picks it up

If you have a /meeting-notes Skill and ask "write up today's call", Claude may apply it automatically based on task type. Convenient, but less reliable when format is critical.

Explicit is safer. Automatic is convenient for everyday tasks where approximate is fine.

Memory & CLAUDE.md Files

There are two separate systems, both layered by scope and stacking on top of each other:

1. CLAUDE.md

What you control

Static

These are the instructions you write for Claude to follow, stored as plain text files at global (~\.claude\CLAUDE.md) or project level (CLAUDE.md in your repo).

For example, telling Claude always to respond in bullet points, never use emojis, etc.

Claude reads all CLAUDE.md files from widest scope to most specific at the start of every session, stacking each layer on top of the last.

Important: CLAUDE.md is context, not enforcement

CLAUDE.md instructions are loaded as context that Claude tries to follow, not as hard rules. Vague or conflicting instructions may be ignored.

2–3 · Global & projectAlways
2

2. Auto Memory System

What you can forget about

Dynamic

What Claude learns and writes itself based on your sessions, stored automatically in ~\.claude\projects\<project>\memory\. You never need to manage this manually.

Scope

Stored on your machine only

Not synced across devices. Each project has its own memory folder.

Control

You can disable it

Set autoMemoryEnabled: false in settings.json to turn it off for a specific project.

💡

Best Practices Tips

End-of-session memory prompt. Before closing the chat, ask Claude: "Save the mistakes you made this session and how I corrected them, so they don't happen again." Claude writes these to Auto Memory and applies them in future sessions.

How Auto Memory is Organised

Claude doesn't use a fixed taxonomy. It creates topic files on its own based on what's worth remembering for that project.

MEMORY.md

The index

A concise table of contents pointing to the topic files. Only the first 200 lines (or 25KB, whichever comes first) auto-load every session, so Claude keeps it short.

Topic files

Named by Claude

Separate .md files Claude creates for specific topics, for example debugging.md, build-commands.md, api-conventions.md. Loaded on demand, only when Claude needs them.

Claude Code
I'm setting up a project called Catalyst — where do I even start?
Great name! Start with a CLAUDE.md — describe Catalyst's goals and architecture so I can help consistently.
What folder structure makes sense?

Hey, can you continue where we left off on Catalyst?
Happy to help! I don't have any context about a project called Catalyst — could you fill me in?
💭
Chats are Isolated.
Memory isn't automatically stored between chats.

No — each chat starts fresh. Conversation history does not carry over.

When you open a new chat, Claude has no memory/knowledge of what was said before. The only things that persist are what you've explicitly asked Claude to save: memory files, CLAUDE.md, and settings for the project and your global.

💡

Best Practices Tips

Change chats frequently to avoid running out of context. Before switching to a new chat, ask Claude to write a summary of what was worked on — including key details and decisions — into its project memory. This acts as a handover note so the next chat can pick up right where you left off.

✓Memory filesFacts Claude saved about you, your preferences, ongoing projects
✓CLAUDE.mdProject instructions, architecture notes, team conventions
✓Skills & settingsAll your configured skills, permissions, and hooks
✗Conversation historyWhat you said and Claude's responses — gone when the chat closes
✗Files Claude readClaude will need to re-read files in the new chat
✗Decisions & reasoningWhy Claude did something — only persists if saved to memory or CLAUDE.md
How to bridge information across chats
Use Memory

Ask Claude to remember key decisions, your preferences, or project context. It will be there in every future chat automatically.

Write to CLAUDE.md

Anything the whole project team or future-you needs to know: architecture decisions, naming conventions, where things live.

Use /export before closing

Exports the full conversation as a text file you can reference or paste into the next session as context.

The Context Window — Claude's Working Memory

The context window is everything Claude can "see" in a single session — it is not persistent storage, just the active working memory for the current chat. Each model has its own limit: Claude Sonnet has a 200k token limit (~150,000 words), while Claude Opus has a larger context window — useful for very long sessions.

You can switch models mid-conversation. If you are running low on context with Sonnet and don't want to compact, switching to Opus lets you continue without interruption. Switching the other way — from Opus back to Sonnet — will force a compaction, since your conversation is likely already over Sonnet's limit.

Claude Chat — Usage & Limits

What are Tokens?

The unit of everything Claude reads and writes

Tokens are the pieces Claude breaks text into — roughly 1 token ≈ ¾ of a word. Every character of your message and Claude's response counts toward your usage limit.

Input Your messages & files
Output Claude's responses

Both count toward your limit. Long conversations and large file uploads consume tokens faster.

Projects

Projects

A dedicated workspace that keeps related conversations, files, and instructions in one place — so Claude already has the context before you type a word.

💡

Memory vs. Files — know the difference

Chat-to-chat memory is not preserved inside a project — each new conversation starts fresh. However, any files you upload at the project level are processed once and made available to every chat in that project automatically. You never have to re-upload a document or wait for Claude to reprocess it — which saves tokens, cuts wait time, and keeps your context clean across the whole engagement.

What it does

Shared context across every chat

Memory files, uploaded documents, and custom instructions are inherited by every chat opened inside the project. Claude knows the background without you having to re-explain it each time.

When to use it

Ongoing work with recurring context

Create a project for each engagement, workstream, or tool you build. Anything you'd otherwise paste into every new chat — a brief, a persona, constraints — belongs in the project instructions instead.

Shared Projects

Projects can be shared with teammates. When you do, they get access to the project-level files, instructions, and context — but your individual conversations stay private to you. No one else can see your chats unless you explicitly share a specific one.

If you share a chat, the recipient gets a read-only snapshot — they can read it but cannot reply, edit, or continue it. Useful for handing off context or showing your work without exposing your full history.

Sharing is restricted to people within your same Accenture licence group (e.g. General Availability 1 or GA2). You cannot share with anyone on a different tier or outside the organisation.

Artifacts

Artifacts

Self-contained outputs that Claude generates alongside your conversation — code, documents, diagrams, and interactive content that live in their own panel and can be iterated on, exported, or published.

💡 Tip — Artifacts are best for outputs you'll reuse, share, or continue editing. For a quick one-off answer, a normal chat response is fine — don't force everything into an artifact.

Types of Artifacts

Code, docs, visuals & more

Runnable code snippets, full HTML pages, Markdown documents, SVG graphics, Mermaid diagrams, and React components. Claude picks the right type automatically based on what you ask for.

Iterating

No more downloading a new file every time

Think of it like Claude Code for chat — instead of downloading an updated HTML after every change, the artifact lives in the side panel and updates in place. Everyone sees the latest version instantly, and you keep refining it right there without touching a file.

Sharing

Publish with a link

Artifacts can be published and shared via a public URL — useful for sharing interactive tools, reference pages, or lightweight deliverables with teammates without sending a file.

⚠️ Same licence group only. The link is only accessible to people within your Accenture licence group — e.g. if you're on General Availability 1 or GA2, only others on the same tier can open it.

Styles

Styles let you control Claude's tone, format, and communication approach for a conversation — without having to explain your preferences every time.

Instructions for Claude

Settings → General → Instructions for Claude

A custom system prompt applied to every chat. Tell Claude who you are, how you work, and how to respond — so you never have to re-explain in every conversation. We've put together a relatively generic structure below that you can use as a base and adapt to your own context.

⚙️ Tune it like a system prompt — be specific about role, output style, and what to skip. Vague instructions ("be helpful") do nothing. Precise instructions ("lead with the recommendation, skip caveats") change every response.

Suggested instructions

I'm a <Role> at Accenture, working within the Supply Chain & Engineering Team. My work spans Agentic AI implementations, Data Foundation Transformations, internal tooling, and client-facing strategy deliverables. My clients are typically <industry and size> navigating AI adoption and/or data transformation.

Be direct and structured: lead with the recommendation, then the reasoning. No preamble, no pleasantries, no "great question." Don't explain what you're about to do, just do it. Skip caveats unless they materially change the answer.

When I ask strategic questions, respond like a senior consultant: give me a point of view, not a menu of options. If my approach is wrong, tell me. I'd rather be challenged than validated.

I often work in PowerPoint, HTML, and <Others>. Outputs should be client-ready by default: clean, precise, no filler. When I ask for a framework or analysis, structure it so I could put it in the desired format without rewriting it.
